Contract example

A simplified AGENT.agent file. The real contract supports workflows, tasks, skills, permissions, safety, validation, and output requirements.

AGENT.agent
project:
name: my-project
purpose: "Describe what this repo does"
safe_commands:
- cargo test
- cargo fmt -- --check
sensitive_paths:
- src/auth/**
- .github/workflows/**
approval_required:
- destructive_commands
- release_files
- auth_logic

How AgentML compares

AgentML does not replace all project docs. It gives agents a structured, validateable contract that can be checked locally and in CI.

Tool / Format Best for Limitation Where AgentML fits
README.md Human onboarding Unstructured; agents can ignore it Complements with enforceability
AGENTS.md Loose agent instructions No validation or enforcement Adds structure and risk scoring
CLAUDE.md Model-specific guidance Tied to one model/tool Portable across agents
Cursor rules Editor-specific guidance Editor-locked; no CLI or CI CLI, CI, and skill system
AgentML Agent execution contracts YAML MVP; no runtime yet Validateable, enforceable, portable

FAQ

Is AgentML an agent runtime?

No. AgentML is a contract layer. It defines project rules, context, skills, validation, and risk signals for coding agents.

Does AgentML sandbox agents?

No. It is not a sandbox. It helps define and validate boundaries, but execution safety still depends on the agent, user, and environment.

Does it replace AGENTS.md?

Not necessarily. AGENTS.md is useful instruction text. AgentML adds structure, validation, risk scoring, skills, and CI enforcement.

Which agents can use it?

Any coding agent that can read files or use generated context. Cursor, Claude, ChatGPT, Windsurf, custom agents, and MCP-based systems can all consume the generated context.

Why YAML right now?

YAML is the MVP format because it is readable, familiar, and easy to validate. A native AgentML syntax can come later if real usage proves the need.
